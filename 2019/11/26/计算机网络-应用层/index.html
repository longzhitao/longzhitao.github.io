应用层协议原理

网络应用体系结构

客户-服务器体系结构（C/S）

在客户-服务器体系结构（client-sever architecture）中，有一个不会关闭的主机叫做服务器，它负责数据的管理，服务于多个称为客户的主机，而称为客户的主机负责完成与用户的交互任务。

如Web应用程序，当我们浏览网页的时候，我们的浏览器会向远方的Web服务器发送请求，当Web服务器接收到我们的请求的时候，它便会向发送我们所请求的对象作为响应。

注意

  • 客户之间不直接通信
  • 服务器拥有固定的IP地址

P2P体系结构

在P2P体系结构（PSP architecture）中，每台主机都称为对等方（Peer），应用程序可以在这些间断连接的主机直接通信，在通讯中每个对等方既可以是客户，也可以是服务商（数据资源的提供者）。

如迅雷、百度网盘，用户可以在一台主机上上传资源，然后再另一台主机上下载上传的资源。

进程通信

客户和服务器进程

运行再多个端系统上的程序，进行通信的并不是程序，而是进程（ 进程是资源（CPU、内存等）分配的基本单位，它是程序执行时的一个实例，即运行中的程序 ），两个不同端系统上的进程，通过跨越计算机网络交换报文（message）进行通信。

在每对通讯进程中，我们通常把两个进程之一标记为客户（client），另一个标记为服务器（server）。对于Web而言，浏览器是一个客户进程，Web服务器是一台服务器进程。

在P2P文件共享的某些应用中，一个进程既可以是客户，也可以是服务器。一般发起通信的进程被标识为客户，在会话开始说等待联系的进程是服务器。

进程之间的接口

进程通过一个称为套接字（socket）的软件接口向网络发送报文和从网络接收报文。

套接字是建立网络应用程序的可编程接口，套接字又称为应用程序和网络之间的应用程序编程接口（Application Programming Interface API）。

进程寻址

主机上的进程向另一个主机上的进程发送分组，必须要知道另一个主机的地址，为了标识目的地，我们需要定义两种信息：

  • 主机的地址——IP地址（IP address）
  • 进程的标识符，目的地——端口号（port number）

应用程序服务要求分类

可靠数据传输

如邮件、文件传输和金融应用，数据丢失会照成灾难性后果，就必须为这些需要可靠数据传输的应用定义一个能提供可靠数据传输的协议，能容忍数据丢失的应用称为容忍丢失的应用。

吞吐量

如多媒体应用，吞吐量低的话会使得多媒体的编码速率低，即我们观看视频的分辨率低，这是难以忍受的，这些需要一定吞吐量的应用称为带宽敏感应用

定时

如网络游戏，我们对传输数据的交付时间有着严格的限制，如果时间长，对应的游戏就会卡顿，这类应用对时延非常敏感。

安全性

如邮件我们希望里面的内容只被接收方阅读，这时候我们就需要运输服务能保证安全性，只有在接收方接收到数据后才能进行解密获取数据内容。

因特网提供的运输服务

TCP服务

  1. 面向连接的服务
  2. 可靠的数据传送服务

UDP服务

  1. 无连接的最小服务

应用层协议

应用层协议（application-layer protocol）定义了运行在不同端系统上的应用程序进程如何相互传递报文。特别应用层定义了：

  • 交换报文的类型，例如请求报文和响应报文
  • 各种报文类型的语法，如报文种的各个字段及这些字段是如何描述的
  • 字段的语义，即这些字段中的信息的含义
  • 确定一个进程何时以及如何发送报文，对报文进行响应规则

应用层协议知识网络应用的一部分。

Web和HTTP

HTTP概况

Web的应用层协议是超文本传输协议（HyperText Transfer Protocol，HTTP）。

Web页面（Web page）（也叫文档）是由对象组成的，一个对象只是一个文件，如一个HTML文件、一个JPEG图像、一个Java小程序或是一个视频片段。

HTTP使用TCP作为它的支撑运输协议。

HTTP是一个无状态协议（stateless protocol），因为HTTP不会保存关于客户的任何信息。

非持续连接和持续连接

《如何优雅的谈论HTTP／1.0／1.1／2.0》

HTTP1.0

HTTP1.0是HTTP协议最遭的标准。为了提高系统的效率，HTTP1.0规定浏览器和服务器只保持短暂的连接，浏览器的每次请求都要与服务器建立一个TCP连接，服务器完成请求处理后会立即断开TCP连接，服务器不会跟踪每一个客户，也不记录过去的请求。如果浏览器访问一个有8个JPEG图形以及一个HTML基本文件的Web网页，会产生9个TCP连接，每个TPC连接只传输一个请求报文和回应报文。如果图形文件很小，每次连接只是传输一个文档，那么建立和关闭TCP连接就是一个比较费时的过程。

HTTP1.1

为了克服HTTP 1.0的这个缺陷，HTTP 1.1支持持久连接（HTTP/1.1的默认模式使用带流水线的持久连接），在一个TCP连接上可以传送多个HTTP请求和响应，减少了建立和关闭连接的消耗和延迟。一个包含有许多图像的网页文件的多个请求和应答可以在一个连接中传输，但每个单独的网页文件的请求和应答仍然需要使用各自的连接。HTTP1.1还允许客户端不用等待上一次请求结果返回，就可以发出下一次请求，但服务器端必须按照接收到客户端请求的先后顺序依次回送响应结果，以保证客户端能够区分出每次请求的响应内容，这样也显著地减少了整个下载过程所需要的时间。

HTTP2.0

HTTP2.0比先前的协议在性能上有着很大的提升。

HTTP2.0与之前的协议对比

HTTP2.0增加了以下功能

  • 多路复用
  • 二进制分帧
  • 首部压缩（Header Compression）
  • 服务端推送（Server Push）

HTTP报文格式

HTTP报文有两种：请求报文和响应报文

HTTP请求报文

典型的HTTP请求报文例子：

GET /somedir/page.html HTTP/1.1
Host: www.someschool.edu
Connection: close
User-agent: Mozilla/5.0
Accept-language: fr

HTTP请求报文主要由请求行、首部行（请求头部）、请求正文3部分组成

请求行

由3部分（3个字段）组成：方法字段（请求方法）、URL字段和HTTP版本字段

方法字段 )包括GET、POST、HEAD、PUT和DELETE。

HTTP版本字段的格式为：HTTP/主版本号.次版本号（HTTP/1.0）
